# Data

Each problem keeps its instances in `problems/<slug>/instances/`. The problem's
`config.py` (`load_data`) parses these files into instance dicts; `spec.py`
wraps that loader and adds the feasibility/objective tools.

## The 7109-instance set

The repository ships the **exact instance set evaluated in the paper: 7109
instances across the 40 problems**, in the original OR-Library file formats.
Nothing is sampled or omitted — the repo is self-contained.

Many classical problems pack **multiple instances per file** (one file holds
several cases). An instance is therefore identified by `(file, case_idx)`, and
the framework addresses a single case as `path::idx`. So the on-disk file count
is smaller than the instance count (e.g. `hybrid_reentrant_shop_scheduling`:
27 files → 1350 instances; `travelling_salesman_problem`: 3 files → 272).

The authoritative manifest is **`instances.json`** at the repo root:

```json
{ "<slug>": [ {"file": "<relpath under instances/>", "case_idx": 0, "split": "train"}, ... ] }
```

Verify the repo supports exactly these 7109:

```bash
python scripts/verify_instances.py
```

### Notes
- The 4 novel problems are laid out as `instances/train/` (128) +
  `instances/test/` (100) = 228 each, the set their shipped portfolios were
  evolved and tested on. Their instances are generated by `build_instances.py`.
- `maximal_independent_set` keeps three subfolders (`er_test/`, `er_valid/`,
  `er_large_test/`) because some files share a basename across folders with
  different content.
- Two infeasible instances present in earlier data were dropped to match the
  paper: `constrained_non_guillotine_cutting` (650, not 651) and
  `resource_constrained_shortest_path` (23, not 24).

The classical CO-Bench instances originate from the third-party
[CO-Bench](https://arxiv.org/abs/2504.04310) benchmark.
